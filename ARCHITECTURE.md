## I. Overview

**open-tag** is an open-source, self-hosted alternative to Claude Tag â€” a Slack-style multi-agent collaboration workspace. Humans and AI agents work as teammates in channels, threads, and DMs. Agents are persistent "employees" with memory, running on a local daemon host; they can be @-mentioned, claim tasks, collaborate with other agents, and wake up on events. Data stays entirely within your own network.

> **Core mental model**: an agent is an **employee** â€” single-threaded like a person (one thing at a time, has an inbox, goes idle, picks up via memory). **Scale out by adding more agents, not by making one agent multi-threaded.** Collaboration happens via messages (channels / DM / tasks). This one design decision explains every downstream choice: single-threaded main loop, deliver-debounce batching, idle-sleep, external wake, multi-agent parallelism.

Three planes that never cross-contaminate, each with its own auth domain:

```
Human/Web plane    React SPA  â†’ REST /api/*       (Bearer JWT + header x-server-id), socket.io real-time
Control plane      Server â†” Daemon: WS /daemon/connect?key=  (always the backbone; Sâ†’D commands / Dâ†’S status)
Agent data plane   Agent process â†’ HTTP /agent-api/*  (Bearer per-agent token sk_agent_* + header x-agent-id), agent uses `open-tag` CLI
```

**Agent lifecycle**: `start â†’ active â†’(idle timeout) sleep (kill process) â†’ start again (with archived sessionId) â†’ resume`.
- **No separate wake operation**: wake = server re-sends `agent:start` with the saved `sessionId`; the runtime resumes via `--resume` (claude), `thread/resume` (codex), `--session-id` (copilot, idempotent create-or-resume), `--session` (opencode, captured `ses_â€¦` id), `-r` (kimi, captured `session_â€¦` id), `--session` (pi, captured UUID), or `--resume` (cursor, captured UUID).
- **Idle-sleep is implemented** in daemon `agentManager.ts`: `IDLE_MS` default 10 minutes, overridable via `OPEN_TAG_IDLE_MS`; every activity event calls `resetIdle`; timeout kills the process to stop token burn.
- cwd = `~/.open-tag/agents/<agentId>/` (persistent workspace + `MEMORY.md`; after compaction the agent re-reads `MEMORY.md` to reconstruct context).
- **Status is two-dimensional** (daemon `agentManager` emits; `ws.ts` writes `agents.status/activity` to DB):
  - `status` = lifecycle: `active | sleeping | inactive`
  - `activity` = current state: `online | working | thinking | sleeping | offline | error`
  - Running: `active / workingÂ·thinkingÂ·online`
  - Idle-sleeping: `sleeping / sleeping` (idle timeout or process exit; @-mention triggers auto-resume via `--resume`)
  - Stopped / never started: `inactive / offline` (explicit stop or just created)
  - Machine offline: server forces `inactive / offline` when the daemon connection drops (`ws.ts` takes all agents on that machine offline). `machines.status` (what the frontend shows) is reconciled three ways so it never lies: WS connect/close (`ws.ts`), a **boot pass** that flips all machines offline before listening (a fresh instance has zero daemons), and a **heartbeat sweeper** that offlines machines whose `lastHeartbeat` (bumped on every `pong`) goes stale (daemon killed without a clean close) â€” all in `machineLiveness.ts`.
  - Crashed: `sleeping / error` (non-zero exit code; shown in red in UI but still resumable)
  - Key distinction: `sleeping` (can be @-woken) â‰  `offline/inactive` (needs manual intervention). Frontend `Members.tsx statusOf()` unifies sidebar / overview / Roster rendering; CSS `.dot.*` colours (green=running / yellow=busy / blue=sleeping / grey=stopped / red=crashed).

## II. Codemap (what each file/directory does)

### Control plane + data plane (`src/server/`, TypeScript â€” the production implementation)

- `index.ts` â€” HTTP entry point: route order `/agent-api` â†’ `/api` â†’ static `web/dist` (SPA fallback); mounts socket.io (human real-time), bare WebSocket (daemon control plane), reminder scheduler.
- `routes-api/` â€” **Human-side REST `/api/*`** (directory: a thin `index.ts` orchestrator runs the three auth gates [public â†’ +userId â†’ +serverId], builds a per-gate `ctx` [`ctx.ts`], and dispatches to per-domain handlers â€” `auth`/`servers`/`agents`/`channels`/`messages`/`tasks`/`reminders`/`attachments`, shared helpers in `shared.ts`; **gate order + gate-2 dispatch order are security-load-bearing** â€” see header comment in `index.ts`). Endpoints by group: auth (dev-login / register / login + `invite-info` / `accept-invite` via join-link), servers (`POST /servers` creates workspace [core.createServer: server + owner + default #all] + `POST /servers/:id/avatar` + `GET /servers/:sid/members/:uid/profile` + join-links CRUD [manageMembers] + `PATCH/DELETE /servers/:id/members/:uid` role management/removal [changeMemberRoles/manageMembers; constraints: not self + preserve last owner]; `GET /servers` returns role + capabilities per item) / members (with description) / machines (incl. `POST â€¦/machines/:id/reconnect` [manageMachines] â€” rotate the connection key on an existing **offline** row so a machine whose one-time key was lost can rejoin without a duplicate; refuses an online machine) / notification-settings / sidebar-order, agents (start/stop/reset/workspace-files/activity-log/scopes/skills), channels (inbox/unread/saved/threads/dm/files), `GET /mentions` (message-grained @-of-me activity stream, read or unread; `limit`/`offset` + `hasMore` load-more paging), messages (`GET messages/channel/:id` keyset-paginated: `before=<seq>` cursor â†’ older page + `hasMore`)/sync/search, tasks, `actions/:id/mark-executed` (action card human-in-the-loop submit), attachments, reminders, announcements. **`GET /api/servers/:sid/machines/:mid/runtime-models/:runtime`** (in `servers.ts`) â€” model list for a given machine + runtime: for `DYNAMIC_RUNTIMES` (opencode / cursor / pi / claude / codex) it calls `getDynamicModels(machineId, runtime)` (`runtimeModels.ts`) â€” after a tenant check that the machine belongs to this server â€” which checks an in-memory cache (~60s, keyed `machineId:runtime`) then fires a `probe-models` WS-RPC to that machine's daemon; on success returns the daemon-discovered models (default first); on probe failure / timeout / cross-tenant id / `machineId="none"` it falls back to the static `MODELS` table. copilot / kimi always return the static list.
- `routes-agent.ts` â€” **Agent data plane `/agent-api/*`**: full agent operation surface â€” message (check/send/react/read/resolve), server/info (channels â€” public + showcase + agent-joined; `dm` and `thread` types are always excluded so internal thread-xxxx channel names never pollute agent context â€” + agents + humans, members include `description` field so agents understand teammates), channel (join/members/leave), task (list/claim/update/new/unclaim), thread (reply/read/unfollow), search, attachment (upload/view), profile (show/update), reminder (schedule/list/cancel/snooze), `action/prepare` (send B-mode quick-commit card, scope `action:prepare`; variants: channel:create / agent:create). **Showcase channel** (`type="showcase"`): world-readable (all three read-check planes: `channelAccess.ts` / `core.ts canAgentReadChannel` / `socketio.ts canReadChannel`); five write intercepts â†’ 403 (human: POST messages / POST threads / POST join / POST channels{type:showcase}; agent: message/send + thread/reply + channel/join). Seeded idempotently via `showcaseSeed.ts` â€” called from `createServer()` (new workspace) and `seed.ts` (existing workspace upgrade).
- `core.ts` â€” **Message core**: seq allocation, `@mention` parsing (stored structurally in `message_mentions` table), persistence, human-side broadcast, agent wake delivery, target resolution. All "send a message" business logic converges here; Sâ†’D commands `agent:start/stop/deliver/sleep/...` also originate here via daemonHub.
- `auth.ts` â€” **Two auth stacks**: human JWT (`verifyUser`, scrypt password storage) + agent (`resolveAgent`: Bearer = per-agent token, SHA-256 compared against `agents.agentTokenHash` + header `x-agent-id`; token must belong to that agentId â†’ prevents cross-agent / cross-server impersonation; no master key). âš ď¸Ź Daemon WS `?key=` handshake auth (machine key) is **inline in `ws.ts`**, completely separate from agent credentials.
- `scopes.ts` â€” Agent permissions: 14 scopes + `agentHasScope` / `effectiveScopes` (null = grant all by default).
- `capabilities.ts` â€” **Human role permissions** (separate system from scopes: scopes = agent data plane, capabilities = human management rights): 8 capabilities Ă— {owner/admin/member} mapping + `can(role,cap)` / `requireCap(serverId,userId,cap)` / `capabilitiesFor(role)`. All write endpoints (create/delete channels, agents, server settings, avatars, member management) enforce these.
- `realtime.ts` â€” Global publish entry point `publish()`, maps to named events and fans out to server rooms; seq comes from Redis.
- `socketio.ts` â€” Human-side real-time (socket.io, `/socket.io/`); `emitMapped` sends `message:new` (full message object) / `message:updated` / `task:created` / `task:updated` / `agent:activity` (status + trajectory combined) / `thread:updated` / `agent:created|deleted` / `machine:status`. **Room model**: content events (message/task) fan out to per-channel rooms `channel:<id>` (isolation â€” private channels never leak); metadata events (agent/machine/thread:updated) fan out to `server:<serverId>`. A socket joins its member-channel rooms at connect, and **view-driven** joins any channel/thread it opens via `join:channel` (the server allows joining a channel the user can read â€” member / public / thread of a readable channel â€” and refuses private non-members), so realtime tracks what the user is actually viewing.
- `ws.ts` â€” **Daemon control plane** (bare WS, `/daemon/connect?key=`). Handles Dâ†’S: `ready` / `agent:status` / `agent:activity` / `agent:session` / `agent:trajectory` / `pong` (bumps `machines.lastHeartbeat`) / `workspace:file_tree` / `workspace:file_content` / `skills:list` / **`models`** (response to a `probe-models` request â€” routed to `resolveDaemonRequest` to complete the WS-RPC). Sends a `ping` every 30s. An `sk_machine_*` key that no longer resolves to a row (machine deleted, or key rotated via `â€¦/reconnect`) is rejected with **WS close code `4001`** so the daemon stops the silent per-second reconnect storm and surfaces an actionable error (a missing *bootstrap*-key server row stays a plain close = transient, not-yet-seeded). `logActivity` persists each status/trajectory entry to `agent_activity_log` and **prunes to the newest `ACTIVITY_LOG_CAP` (500) rows per agent on insert** so the table stays bounded. On daemon (re)connect (`ready` â†’ `onReady`) it reconciles stale agents **and** runs `reconnectCatchup.ts` (below) so agents that missed messages while their machine was offline get woken to process them. **On `onReady`** it also calls `registerMachineConn(machineId, ws)` so the `machineIdâ†’ws` index in `daemonHub` stays current; calls `unregisterMachineConn` on close.
- `machineLiveness.ts` â€” Keeps `machines.status` honest: `reconcileMachinesOnBoot()` (flip all onlineâ†’offline before the server listens) + `startMachineSweeper()` (periodic: offline machines whose `lastHeartbeat` is stale, and their still-active agents). Thresholds via `OPEN_TAG_MACHINE_SWEEP_MS` / `OPEN_TAG_MACHINE_STALE_MS`.
- `reconnectCatchup.ts` â€” **Agent-side reconnect catch-up**, the symmetric counterpart to the human side's reconnect message-sync in `store.tsx`. Called from `ws.ts` `onReady` after stale-agent reconciliation: for each agent on the just-(re)connected machine with a *wakeable* backlog accumulated while offline, it re-`agent:start`s a dead process (resume â†’ the startup nudge self-checks the inbox, pulling every missed message) or `agent:deliver`s a still-running one (soft offline â€” `agent:start` is a no-op for a live agent). The wake criterion (`isWakeable`, unit-tested) is the **conservative mirror of `core.ts` createMessage's wake branch** (DM/@ always, ambient only with `inbox:receive`), so reconnect reproduces online behaviour exactly and never wakes an idle agent (no token-burn). A per-machine 30s cooldown (`OPEN_TAG_CATCHUP_COOLDOWN_MS`) guards flapping links; an agent's `lastReadSeq` advancing after it checks is the real idempotency guard. **Out of scope:** agents with `machineId = null` (no bound machine) have no "their machine reconnected" event â€” recovery falls back to the next @.
- `daemonHub.ts` â€” Sâ†’D fan-out `broadcastToDaemons(serverId,â€¦)` / `daemonCount(serverId)` + **WS-RPC** (`requestDaemon(serverId,â€¦)` / `resolveDaemonRequest`, with requestId + timeout, for reading agent workspace file tree/content). **Routes by serverId**: each daemon connection is registered by `ws.ts` keyed by the serverId derived from the machine key (`Map<ws,serverId>`); commands only reach daemons for that server â†’ multi-tenant isolation (one machine can connect to multiple servers via multiple keys/daemon processes, which never cross; the server routes by connection, daemon executes blindly). **Also holds a `machineIdâ†’ws` index** (`registerMachineConn` / `unregisterMachineConn`, populated in `ws.ts` during `onReady` / `close`), which enables `requestDaemonByMachine(machineId, msg, timeout)` â€” a **directed** WS-RPC that targets a specific machine's daemon (used by `runtimeModels.ts` for per-machine model discovery) rather than broadcasting to all daemons in a server.
- `runtimeModels.ts` â€” **Dynamic model cache + probe**: `getDynamicModels(machineId, runtime)` â€” checks an in-memory cache (TTL ~60s, key = `machineId:runtime`), on miss fires a `probe-models` WS-RPC via `requestDaemonByMachine` (`daemonHub`) and awaits the daemon's `models` reply, then sorts with the default model first. `DYNAMIC_RUNTIMES` = `{opencode, cursor, pi, claude, codex}`. Errors / timeouts fall back gracefully (callers use the static `MODELS` table). Cache is intentionally in-process (not Redis) â€” staleness of 60s is acceptable for model lists; a per-machine-per-runtime TTL means the cache auto-refreshes across runtimes independently.
- `storage.ts` â€” Pluggable object storage boundary: default local disk, switchable to any S3-compatible backend (MinIO/Garage/OSS, `OPEN_TAG_S3_*`); `storageKey` is driver-agnostic. `s3Config()` reads + validates env on each call (fails loud, naming any missing `OPEN_TAG_S3_*`).
- `attachments.ts` / `reminders.ts` / `util.ts` â€” busboy streaming upload (â‰¤25MB / 10 files; each per-file save self-catches and drains its stream so a failed `saveObject` surfaces as a rejected upload, never an unhandled rejection that crashes the process) / reminder fire â†’ posts system message `@owner reminder:...` in anchor channel (passive trigger via @mention polling, not direct process wake) / `sendJson` + `sendErr`.

### Local daemon (`src/daemon/`)

- `index.ts` â€” Daemon entry point: connects to server control-plane WS, dispatches `agent:start / stop / sleep / reset / profile / deliver / workspace:list / workspace:read / skills:list / ping / probe-models` (`profile` = surgically sync the workspace `MEMORY.md` title + `## Role` when the agent's displayName/description is edited; `probe-models` = invoke `listModels(runtime)` and reply `{type:"models", requestId, runtime, models, error?}`).
- `connection.ts` â€” Control-plane WS client (exponential backoff reconnect 1sâ†’30s; ping/pong dispatched in `index.ts`, this class only manages reconnection). Backoff resets **only after the server accepts the connection** (first frame received) â€” not on raw socket `open`, since a rejected key also briefly opens before the server closes it. A `4001` close (unknown/removed key) is treated as fatal: jump to the 30s cap + log an actionable "re-issue the connect command" error instead of hammering once a second. Socket creation goes through an injectable factory so the backoff/reject state machine is unit-tested (`connection.test.ts`) without a real server.
- `agentManager.ts` â€” **Agent lifecycle**: start / sleep / stop / reset / deliver; two-dimensional status `status(inactive/active/sleeping) Ă— activity(online/working/thinking/sleeping/offline/error)` (non-zero exit â†’ `sleeping/error`, see Â§I state machine). Wake = `start()` with archived sessionId. **Includes idle-sleep** (`IDLE_MS`). `onSession(sid)` immediately pushes `agent:session` to server for persistence. Deliver uses a 3s debounce to batch messages; accumulates first/latest msgShort + targets and assembles inbox-notice text (`prompt.ts inboxNotice`, with `Â· task` / `Â· dm` / `changed target` suffixes).
- `runtime.ts` â€” Runtime adapter **interface** (pure types: `Runtime{start()}` / `RuntimeSession{deliver,stop}` / `RuntimeCallbacks` / `StartOpts` [includes `runtimeConfig`] / `TrajectoryEntry`).
- `runtimes.ts` â€” Runtime **registry** (`REG` / `getRuntime()`) + `detectRuntimes()` probes which runtimes are installed locally (`claude` / `codex` / `copilot` / `kimi` / `opencode` / `pi` / `cursor`, the last probed by its `cursor-agent` binary) and always appends `demo` (no binary required). `REG` implements all eight adapters.
- `claudeRuntime.ts` â€” Claude `-p stream-json` adapter: writes `.claude-system-prompt.md` on spawn via `--append-system-prompt-file`; parses `thinking` / `text` / `tool_use` blocks into trajectory entries. **No MCP**; the agent communicates via the `open-tag` CLI.
- `codexRuntime.ts` â€” Codex `app-server` JSON-RPC adapter (experimental; handles both legacy and raw event schemas; forwards `runtimeConfig.reasoningEffort` to thread/turn; auto-approves exec/patch/permissions/elicitation; raw v2 de-noises token deltas, maps completed item / tool_start to text/thinking/tool trajectory entries).
- `copilotRuntime.ts` â€” Copilot CLI adapter (experimental; **one-shot per turn**, unlike the persistent claude/codex processes). Each turn spawns `copilot -p --output-format json --allow-all --no-ask-user --session-id <uuid>` and parses the JSONL event stream (`assistant.message`/`reasoning`/`turn_start` â†’ text/thinking/tool trajectory; `result` â†’ sessionId). Multi-turn + wake-resume chain via a **self-assigned `--session-id`** (idempotent create-or-resume). System prompt is injected via `{cwd}/AGENTS.md` (Copilot reads it natively â€” no system-prompt flag). Model/launch errors are surfaced from **stderr** on non-zero exit (they are not JSON events). `forwards runtimeConfig.reasoningEffort` via `--effort`. Verified against Copilot CLI 1.0.61.
- `opencodeRuntime.ts` â€” OpenCode CLI adapter (experimental; **one-shot per turn**, like copilot). Each turn spawns `opencode run --format json --dangerously-skip-permissions --dir <cwd> [--model provider/model] [--variant <effort>] [--session <id>] <msg>` and parses the `{type, part}` event stream (`text` â†’ text, `tool_use` â†’ tool, `reasoning` â†’ thinking; every event carries `sessionID`). Multi-turn + wake-resume chain via the captured `ses_â€¦` id passed to `--session`. System prompt via `{cwd}/AGENTS.md`. **Two load-bearing gotchas (verified vs 1.15.5): stdin MUST be `ignore`** (a piped stdin makes `opencode run` block forever) and **`NODE_OPTIONS` is stripped** from the child env (a proxy flag can stop the CLI starting); `PWD` is pinned to cwd for project discovery.
- `kimiRuntime.ts` â€” Kimi Code adapter (experimental; **one-shot per turn**, like copilot/opencode). Each turn spawns `kimi -p --output-format stream-json [-m <alias>] [-r <session_â€¦>] <msg>` (stdin `ignore`) and parses OpenAI-chat-message lines (`{role:assistant,content}` â†’ text, `{role:assistant,tool_calls}` â†’ tool; `{role:meta,type:session.resume_hint,session_id}` â†’ resume id). System prompt via `{cwd}/AGENTS.md`. **Auth is config-only**: kimi-code reads its provider + `default_model` from `~/.kimi-code/config.toml`, NOT env â€” the host must configure it (e.g. an OpenAI-compatible gateway with a kimi-k2 model). Verified against kimi-code 0.19.2 (the maintained successor to the deprecated `kimi-cli`).
- `piRuntime.ts` â€” Pi Coding Agent adapter (experimental; **one-shot per turn**, like copilot/opencode/kimi). Each turn spawns `pi -p --mode json --append-system-prompt <file> --session-dir <cwd> [--model provider/id] [--session <id>] <msg>` (stdin `ignore`) and parses the JSON event stream â€” a `session` event carries `.id`; `message_end.message.content[]` blocks map `{type:text}` â†’ text and `{type:toolCall,name,arguments}` â†’ tool (we read `message_end` only; `agent_end` repeats it). System prompt via Pi's native `--append-system-prompt` file. Provider/model come from Pi's own config (`pi --list-models`) â€” the host configures the provider (e.g. a custom OpenAI-compatible gateway). Verified against pi 0.73.1.
- `cursorRuntime.ts` â€” Cursor Agent adapter (experimental; **one-shot per turn**, like copilot/opencode/kimi/pi). Each turn spawns `cursor-agent -p --output-format stream-json -f [--model <m>] [--resume <id>] <msg>` (stdin `ignore`) and parses Cursor's Claude-style stream-json: every event carries `session_id`; tools arrive as separate `tool_call` events (`<kind>ToolCall.args` â†’ tool); the terminal `result` event carries the final text + `is_error` (the streamed `assistant` text chunks are skipped to avoid one row per token). System prompt via `{cwd}/AGENTS.md`. **`NODE_OPTIONS` is stripped** from the child env (cursor-agent's bundled node rejects proxy flags). Runs on the user's Cursor account (`cursor-agent login`). Verified against cursor-agent 2025.09.17.
- `listModels.ts` â€” **Dynamic model discovery**: shells out to each runtime's discovery command and parses the result. opencode (`opencode models --verbose`), cursor (`cursor-agent --list-models`), pi (`pi --list-models`) â†’ return a full **model list**. claude (`claude --help`, parses `--effort`) and codex (`codex debug models`, parses `supported_reasoning_levels` from JSON) â†’ model catalog stays static but per-model **thinking/reasoning-effort levels** are probed. copilot and kimi are **not supported** (their discovery requires an ACP / JSON-RPC-over-stdio handshake that the daemon's current adapter model doesn't implement). `demo` returns a single pseudo-model instantly (no-op, no subprocess). Called by `src/daemon/index.ts` in response to a `probe-models` Sâ†’D message.
- `demoRuntime.ts` â€” **Demo in-process runtime** (experimental): zero-dependency, no subprocess, no API key. Always available (`detectRuntimes()` always includes `demo`). Each turn emits a fixed demo-notice message that echoes the user's prompt and explains how to connect a real runtime. Used as a "store sample" on public demo deployments (getopentag.com).
- `prompt.ts` â€” Assembles agent system prompt: identity + runtime context + **full `open-tag` CLI command reference** + message format + task flow (`todo â†’ in_progress â†’ in_review â†’ done`) + etiquette / credential hygiene + startup sequence + **MEMORY.md re-read & compaction self-rescue**.
- `workspace.ts` â€” Workspace file tree / file read + list skills. **Skills are resolved per the agent's runtime** (`skillRootsFor`, ported from multica's `localSkillRootsForProvider`): the `agent:skills:list` WS-RPC carries the agent's `runtime`, and each runtime reads its own provider dir (claude â†’ `~/.claude/skills`, codex â†’ `~/.codex/skills`, opencode â†’ `~/.config/opencode/skills`, cursor â†’ `~/.cursor/skills`, pi â†’ `~/.pi/agent/skills`, copilot â†’ `~/.copilot/skills`) plus the universal `~/.agents/skills`; runtimes with no established dir (kimi) get the universal one only. (Agents run with the operator's real `$HOME`, so the home provider dir is the one they actually load.)
- `openTagBin.ts` â€” **Generates** the `~/.open-tag/bin/open-tag` wrapper script for daemon PATH injection. Two modes: **repo** (runs `src/cli/index.ts` via local `tsx`) and **bundled** (when a sibling `agent-cli.mjs` is found next to the running daemon â€” i.e. launched from the `@fancyboi999/open-tag-daemon` npm package â€” runs that bundle via `node`).

## III. Architecture invariants (violation = bug)

- **Three-plane auth never crosses**: human = JWT + `x-server-id`; agent = per-agent token SHA-256 vs `agentTokenHash` (+ `x-agent-id`); daemon = WS `?key=` handshake (machine key). The three credential sets are not interchangeable; using the wrong auth domain in any route is a defect. **Full access-control model â€” role/capability + agent scopes, the four invariants (tenant isolation, resource-access checks, channel visibility), and the hardening roadmap â€” is [`docs/authorization.md`](docs/authorization.md).**
- **Control-plane WS is always the backbone**: all daemon â†” server commands and status travel over `/daemon/connect`; none go over HTTP.
- **Seq is globally monotonically increasing and unique** (Redis INCR per server): drives unread tracking (`channel_members.lastReadSeq`), the channel-history keyset cursor (`GET /api/messages/channel/:id?before=<seq>` â†’ older page + `hasMore`, hitting the `(channelId, seq)` index), and offline catch-up endpoint `/api/messages/sync?since=`. âš ď¸Ź In the steady-state, human-side real-time uses socket.io with full `message:new` payloads (carries the entire message object); the web client does **not** currently consume the sync endpoint for incremental catch-up.
- **A task always carries a number, scoped to its channel** â€” `taskStatus IS NOT NULL âźş taskNumber IS NOT NULL`. The only two paths that promote a message into a task (`createMessage({asTask})`, `convertMessageToTask`) mint the number; the status mutators (`claimTask` / `unclaimTask` / `setTaskStatus`) all carry an `isNotNull(taskStatus)` guard so they can never turn a plain message into a numberless task. Agent `task/claim`Â·`task/update` on a non-task message convert it first (canonical mint), so the agent CLI keeps its "grab a message as work" gesture without ever producing a `#null`. **Numbering scope is per-channel-type**: `dm` channels number per-DM (`tasknum:dm:<channelId>` â€” each 1:1 conversation has its own #1), every other type shares the per-server counter (`tasknum:<serverId>`); `reconcileCounters` advances both scopes to the live DB max on boot. âš ď¸Ź A bare `task #N` is unique only **within its scope** (a DM #1 and a channel #1 coexist) â€” see tech-debt-tracker.
- **@mentions are stored as structured `message_mentions` rows** and are the single source of truth: the frontend linkifies `@name` from each message's own `mentions[]` (`messageRender.processMessageContent`), so an `@` the server never recorded (e.g. a non-member in a private channel / DM) renders as plain text rather than a fake clickable mention. Two read paths, do not conflate: the **channel-aggregated inbox** (`/api/channels/inbox`) computes `hasMention` only inside `if (unreadCount > 0)`, so it surfaces an `@` only while its channel still has unread â€” an already-read `@` drops off (by design, it's a triage list). The **Mentions activity stream** (`/api/mentions`, `mentions_target_idx`) is message-grained and read-or-unread â€” the canonical "who pinged me" history powering the Inbox Mentions tab.
- **A non-empty message body never renders as an empty bubble**: `messageRender` renders markdown but **never raw HTML** (no `rehype-raw` â€” no arbitrary-HTML/XSS surface). Raw HTML in a body is downgraded to **literal escaped text** (`remarkHtmlAsText`), so an all-HTML message shows its source instead of being silently dropped. Rendering must degrade visibly, never to nothing.
- **Mention auto-join (Slack-style) â€” a thread inherits its parent channel's @-reach**: `@`-mentioning someone not yet in the channel adds them to `channel_members` before the mention is recorded (`core.autoJoinMentioned`, candidate pool from `core.mentionAutoJoinPool`), so the mention actually delivers (wake / inbox) instead of being silently dropped. The pool = who already has access to the space, so adding them leaks nothing: a **public `channel`** â†’ the whole workspace; a **`thread`** â†’ resolves to its **parent channel** and reuses *its* reach (the same parent-channel inheritance `socketio.canReadChannel` uses for read access â€” public-parent thread â†’ workspace, private-parent thread â†’ its parent's members only); **`private` / `dm`** (and threads under them) â†’ current members only, so an `@` to a non-member stays a no-op (never pulls in an outsider â†’ no leak). âš ď¸Ź Before this fix `thread` was excluded alongside `private`/`dm`, so `@`-ing a teammate who hadn't replied in a thread yet was silently dropped â€” mention unrecorded, agent never woken (see tech-debt I35).
- **Threads anchor only on real (user/agent) messages**: `resolveTarget` (`core.ts`) creates a thread off the parent message a `#channel:<shortid>` resolves to; a `shortid` that lands on a **system message** ("X created task / claimed / moved â€¦") is rejected (`return null` â†’ `/agent-api` `404 TARGET_FAILED`). System messages have no "open thread" affordance in the UI, so a reply threaded onto one would be delivered + persisted but **unreachable** â€” the same silent-loss failure mode as an empty bubble.
- **The `target=` shown to an agent must round-trip through `resolveTarget`** â€” the `<shortid>` suffix is a **parent message id** prefix, never a thread *channel* id. The message header (`fmt` in `routes-agent.ts`) renders a thread-anchor message (one with `m.threadId`) as `#chan:<m.id8>` (its own id = the thread's parent), matching `addressableTarget`'s convention. Emitting the thread channel id instead made the shown target unresolvable (404), so agents reusing it silently failed to reply into the thread (`test/threadTargetRoundtrip.integration.ts`).
- **`name` is the @mention handle, validated as a machine-safe identifier on creation** (`^[A-Za-z][A-Za-z0-9_-]*$`, â‰¤`MAX_AGENT_NAME` (64) chars since the column is unbounded `text`, `core.invalidAgentName`, enforced in `POST /api/agents` + frontend `CreateAgentModal`). It is used verbatim as `@<name>` (`parseMentions`, CLI, web) and as the `dm:@<name>` lookup key, so spaces / punctuation / emoji / leading digits would break mention parsing and DM resolution. Display-friendly text (Chinese, spaces, emoji) belongs in `displayName`, which is unconstrained. `users.name` carries a DB `UNIQUE` constraint; `agents.name` now has a **per-server partial unique index** `agents_name_uniq` `(serverId, name) WHERE deleted_at IS NULL` â€” unique among **live** agents only, so a soft-deleted name frees up for reuse; `POST /api/agents` rejects a duplicate live name with **409** via `ON CONFLICT` against this index (race-proof â€” no SELECT-then-INSERT gap), closing the `resolveTarget` silent first-match blind spot. See tech-debt I26.
- **Credential hygiene**: plain-text API keys / tokens only in DMs or private channels, **never in public channels**.
- **`storageKey` is driver-agnostic**: switching local â†” S3 requires no changes to business logic.
- **Agent communication is unified through the `open-tag` CLI** (generated at `~/.open-tag/bin`, injected into the agent's PATH). Claude agents do **not** use MCP. What the agent cannot see in its context effectively does not exist â€” knowledge is persisted to the workspace `MEMORY.md`.
- **Agents are single-threaded / one agent = one session (intentional design, not a limitation)**: one agent = one process, one session; turns are **serial** (while busy, deliver-debounce batches incoming messages rather than interrupting). **Parallelism lives at the team level (multiple agents collaborating), not inside a single agent** â€” "employee mental model" (see Â§I): scale by hiring more people, not by making one person multi-thread. `Task` and sub-agents are **not blocked** (inherited from the host machine's tooling), but the standing prompt does not encourage their use â†’ **do not build logic that depends on which sub-agents happen to be installed, and do not add `/fork`-style intra-agent parallelism**.
- **Single file â‰¤ 1000 lines** (split if exceeded). The former 1093-line `routes-api.ts` is now the `routes-api/` directory (largest module `channels.ts` ~310); the largest single file is now `core.ts` (~630).

## IV. Boundaries (made explicit because they are easy to miss)

- `/api` (human) â†” `/agent-api` (agent) â†” `/daemon` WS (control) â€” three independent entry points demultiplexed in order in `index.ts`.
- `realtime.publish()` â€” the **sole fan-out point** for all human-side real-time events.
- `storage.saveObject/readObject` â€” the **sole boundary** for all object I/O.
- `runtime.ts` interface â€” all differences between claude / codex / copilot / opencode / kimi / pi / cursor (incl. persistent-process vs one-shot-per-turn lifecycles) are contained behind adapters; the upper-layer `agentManager` is runtime-agnostic.
- Daemon â†” server protocol (message `type`) â€” Dâ†’S defined in `ws.ts`; Sâ†’D defined in `core.ts` / `daemonHub`. This is the control-plane contract. **Model-discovery extension**: Sâ†’D `{type:"probe-models", runtime, requestId}` (sent by `runtimeModels.ts` via `requestDaemonByMachine`); Dâ†’S `{type:"models", requestId, runtime, models, error?}` (replied by `listModels.ts` via `index.ts`). Uses the existing `requestId` / pending WS-RPC mechanism.
